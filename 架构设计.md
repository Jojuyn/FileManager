## 2. 核心需求与约束

### 2.1 核心业务需求（影响架构的关键需求）
- 支持多终端下单（APP/小程序/H5），需统一接口入口  
- 订单状态需实时同步至配送系统（配送员APP需实时显示最新状态）  
- 支持“超时未支付自动取消”“缺货自动退款”等自动化流程  
- 需保存3年订单数据，支持按用户/时间/商品多维度查询  


### 2.2 非功能需求（NFR）
| 类型       | 具体指标（可量化）                     |
|------------|----------------------------------------|
| **性能**   | 下单接口响应≤200ms；查询接口≤500ms<br>峰值TPS≥8000；日均请求量200万 |
| **可用性** | 核心链路（下单-支付-库存）可用性99.99%<br>非核心链路（评价-售后）99.9% |
| **安全性** | 支付信息加密存储（AES-256）<br>接口需签名校验（防篡改）<br>符合《个人信息保护法》（敏感信息脱敏） |
| **可扩展性** | 新增支付方式（如数字人民币）时，无需修改订单核心逻辑<br>支持单模块独立扩容（如库存模块单独加节点） |
| **可维护性** | 模块代码复用率≥60%<br>故障定位时间≤30分钟（通过链路追踪） |


### 2.3 约束条件
- **技术约束**：基于Java（团队栈）、Spring Cloud（公司标准）、K8s部署（基础设施）  
- **成本约束**：服务器预算≤80万；优先使用开源组件（避免商业软件采购）  
- **时间约束**：4个月内完成架构落地；需复用现有库存系统（减少开发量）  
- **合规约束**：支付模块需通过PCI DSS认证；数据存储需符合等保三级  


## 3. 整体架构设计

### 3.1 架构总览
#### 架构图（建议配图：分层/微服务架构图）
![架构总览图](此处插入架构图链接或描述：如“分层架构：接入层→应用层→服务层→数据层”）

#### 核心逻辑说明
- 接入层：统一接收请求，处理认证、限流、路由  
- 应用层：编排业务流程（如“下单流程=创建订单+扣减库存+创建支付单”）  
- 服务层：提供原子能力（如“创建订单”“扣减库存”等独立服务）  
- 数据层：存储与计算（数据库、缓存、消息队列等）  


### 3.2 分层/模块设计（以微服务为例）

| 层级/模块   | 核心职责                          | 核心组件/服务                | 依赖关系                          |
|-------------|-----------------------------------|-----------------------------|-----------------------------------|
| **接入层**  | 流量入口、认证、限流、路由        | API网关（Spring Cloud Gateway）、CDN | 依赖应用层（转发请求）            |
| **应用层**  | 业务流程编排（非原子逻辑）        | 下单流程服务、退款流程服务   | 依赖服务层（调用原子服务）        |
| **服务层**  | 原子能力提供（按领域拆分）        | 订单服务、支付服务、库存服务、配送服务 | 依赖数据层（读写数据）            |
| **数据层**  | 数据存储与中间件                  | MySQL（业务库）、Redis（缓存）、RocketMQ（消息）、Elasticsearch（搜索） | 被服务层调用                      |
| **基础设施层** | 支撑系统运行的通用能力            | 注册中心（Nacos）、配置中心（Apollo）、链路追踪（SkyWalking） | 所有上层模块依赖                  |


### 3.3 核心流程设计（以“下单流程”为例）
#### 流程图（建议配图：时序图/流程图）
![下单流程图](此处插入流程图链接或描述：用户请求→接入层→应用层→服务层→数据层）

#### 流程说明
1. 用户通过APP提交订单请求（含商品ID、数量、收货信息）  
2. 接入层（API网关）校验token、签名，路由至“下单流程服务”  
3. 应用层（下单流程服务）依次调用：  
   - 订单服务：创建订单（状态“待支付”）  
   - 库存服务：扣减库存（加分布式锁防超卖）  
   - 支付服务：创建支付单（生成支付链接）  
4. 数据层：  
   - 订单/库存/支付数据同步写入MySQL  
   - 订单创建事件通过RocketMQ异步发送（供配送/日志服务消费）  
5. 结果返回：支付链接返回至用户APP  


## 4. 技术选型设计

### 4.1 核心技术栈清单
| 层级/场景       | 技术选型          | 选型理由（对应需求/约束）                     |
|-----------------|-------------------|----------------------------------------------|
| 接入层          | Spring Cloud Gateway | 支持动态路由/限流；符合公司微服务标准          |
| 服务框架        | Spring Cloud Alibaba | 提供注册/配置中心；成熟稳定（适配高可用需求）  |
| 数据库（核心）  | MySQL 8.0（主从） | 支持事务（订单一致性）；主从分离提升读性能     |
| 缓存            | Redis Cluster     | 高性能（支持TPS需求）；集群模式（高可用）      |
| 消息队列        | RocketMQ          | 支持事务消息（库存-订单一致性）；团队熟悉      |
| 搜索            | Elasticsearch     | 支持多维度查询（订单搜索需求）；开源免费      |
| 部署容器        | Docker + K8s      | 支持弹性扩缩容（应对峰值流量）；公司基础设施   |
| 链路追踪        | SkyWalking        | 分布式追踪（故障定位需求）；轻量开源          |


### 4.2 关键技术方案（架构难点解决方案）

#### 4.2.1 高可用方案
- 服务高可用：服务多节点部署（至少3节点），注册中心双活（Nacos集群）  
- 数据高可用：MySQL主从复制（1主2从），Redis集群（3主3从），跨机房部署（北京+上海）  
- 故障兜底：核心接口降级策略（如“库存不足时返回‘缺货’，不阻塞下单”）；熔断机制（第三方支付超时则切换备用渠道）  


#### 4.2.2 性能优化方案
- 缓存策略：热点商品库存缓存（Redis）；订单列表缓存（用户维度，10分钟过期）  
- 分库分表：订单表按用户ID哈希分8库，按时间分表（每月1表）（解决3年数据存储问题）  
- 异步化：非核心流程异步处理（如“订单创建后，异步发送通知短信”）  
- 读写分离：查询请求走MySQL从库；写请求走主库  


#### 4.2.3 数据一致性方案
- 强一致性场景（订单-库存）：TCC分布式事务（Try：预扣库存；Confirm：确认扣减；Cancel：回滚）  
- 最终一致性场景（订单-配送）：基于消息队列（RocketMQ事务消息），失败重试（3次+人工介入）  


## 5. 数据架构设计

### 5.1 数据存储设计
#### 存储分层策略
- 热数据（近3个月订单）：MySQL（支持高频读写）  
- 温数据（3个月-1年）：MySQL从库+对象存储（OSS）（读多写少）  
- 冷数据（1年以上）：OSS+归档存储（低成本，支持批量查询）  


#### 数据库设计原则
- 核心库（订单/支付/库存）：MySQL（事务支持）  
- 非结构化数据（订单快照/物流轨迹）：MongoDB（灵活存储）  
- 搜索数据（多维度查询）：Elasticsearch（全文检索）  
- 日志数据：ELK（Elasticsearch+Logstash+Kibana）  


#### 分库分表策略（订单表为例）
- 分库键：user_id（哈希取模，8个库）—— 确保同一用户订单在同一库（便于查询）  
- 分表键：create_time（按月分表）—— 符合时间维度查询需求  
- 路由规则：通过ShardingSphere实现分库分表中间件（自动路由）  


### 5.2 数据流转设计
#### 同步/异步机制
- 同步：核心数据（订单创建、库存扣减）实时写入MySQL  
- 异步：非核心数据（订单日志、统计数据）通过消息队列异步同步  


#### 数据一致性保障
| 场景                 | 一致性要求 | 方案                          |
|----------------------|------------|-------------------------------|
| 订单创建+库存扣减    | 强一致     | TCC分布式事务                 |
| 订单状态变更→配送系统 | 最终一致   | RocketMQ消息（失败重试3次）   |
| 订单数据→统计报表    | 最终一致   | 定时任务（每小时同步一次）    |


## 6. 部署与运维架构

### 6.1 部署架构
#### 部署图（建议配图：机房/节点分布）
![部署架构图](此处插入部署图链接或描述：如“北京+上海双机房，核心服务跨机房部署”）

#### 环境规划
| 环境       | 节点配置（核心服务）       | 用途                          |
|------------|----------------------------|-------------------------------|
| 开发环境   | 2节点（4核8G）             | 开发调试                      |
| 测试环境   | 4节点（8核16G）            | 功能测试、性能测试            |
| 预生产环境 | 8节点（8核16G）            | 模拟生产验证（数据与生产一致） |
| 生产环境   | 16节点（16核32G）          | 线上服务（北京8节点+上海8节点） |


#### 弹性伸缩策略
- 自动伸缩：基于K8s HPA（CPU>70%或内存>80%时，自动增加节点，最大20节点）  
- 手动伸缩：大促前（如618）提前扩容至32节点，结束后缩容  


### 6.2 运维监控设计
#### 监控指标
- 业务指标：订单量、支付成功率、库存可用率  
- 技术指标：接口响应时间、TPS/QPS、错误率、节点CPU/内存/磁盘使用率  
- 链路指标：调用链耗时（各服务节点耗时占比）  


#### 监控工具链
- 指标采集：Prometheus（系统/应用指标）  
- 可视化：Grafana（仪表盘展示）  
- 日志收集：ELK（日志查询与分析）  
- 链路追踪：SkyWalking（分布式调用链）  
- 告警：AlertManager（对接企业微信/邮件）  


#### 告警策略
| 指标                 | 告警阈值               | 告警级别 | 处理方式                          |
|----------------------|------------------------|----------|-----------------------------------|
| 下单接口响应时间     | >500ms（持续1分钟）    | 警告     | 开发群通知（检查缓存/数据库）      |
| 服务错误率           | >1%（持续30秒）        | 严重     | 开发+运维群通知（立即排查）        |
| MySQL主库连接数      | >80%最大连接数         | 警告     | 运维群通知（准备扩容）            |
| 核心链路可用性       | <99.9%（1小时内）      | 紧急     | 触发故障预案（可能需要回滚）      |


## 7. 风险与应对措施

| 风险类型       | 具体风险描述                          | 影响程度 | 应对措施                                      |
|----------------|---------------------------------------|----------|-----------------------------------------------|
| **性能风险**   | 大促峰值TPS可能突破10000（设计8000）  | 高       | 1. 提前扩容至32节点<br>2. 非核心接口降级（如评价功能关闭）<br>3. 队列削峰（请求暂存RocketMQ） |
| **可用性风险** | MySQL主库故障（无自动切换）           | 高       | 1. 部署主从自动切换工具（MGR）<br>2. 备用库实时同步（15秒延迟）<br>3. 故障时手动切从库（预案演练） |
| **技术风险**   | 团队对TCC事务不熟悉（开发效率低）     | 中       | 1. 提前开展培训（3天）<br>2. 引入成熟框架（Seata）<br>3. 先做demo验证（2周内） |
| **成本风险**   | 双机房部署成本超预算（预估100万）     | 中       | 1. 非核心服务（如评价）单机房部署<br>2. 冷数据存储用低成本OSS<br>3. 复用公司闲置服务器 |
| **合规风险**   | 支付模块未通过PCI DSS认证（延期上线） | 高       | 1. 提前6个月启动认证准备<br>2. 聘请第三方咨询机构<br>3. 预留1个月整改时间 |

### 9.2 架构决策记录（ADR）
